## 1. 文档目标

本文档将阶段 A-D1 拆解为可直接进入开发的任务清单，要求每个任务具备：

- 明确的目标
- 归属阶段
- 前后端责任边界
- 依赖关系
- 完成定义

任务粒度按“可在一个开发周期内完成并验收”为原则，不按 Jira 工具格式强绑定。

---

## 2. 使用方式

建议按照以下层级使用本清单：

- `Epic`: 一组可独立验收的业务能力
- `Task`: 可分配给单个开发者的实现项
- `Done`: 进入联调或验收前必须满足的条件

建议执行顺序：

1. 先完成阶段 A 全部任务
2. 阶段 B 与阶段 C 可部分并行，但必须先完成鉴权和主数据
3. 阶段 D1 必须依赖阶段 C 的任务流转接口完成

## 4. 阶段 A 任务拆解

## A1 仓库基线整理

### 目标

清理当前仓库中的 starter 残留，使项目命名、目录职责和运行入口与实际产品一致。

### Tasks

- A1-1 清理或冻结 `backend/apps/web` 示例工程
- A1-2 修正后端 Swagger 标题、描述、端口配置
- A1-3 统一后端 API 全局前缀为 `/api`
- A1-4 整理后端模块目录骨架
- A1-5 更新 `README` 与目录说明，反映当前前后端结构

### Done

- 仓库内不再保留明显误导的示例业务命名
- API 服务具备统一访问入口

## A2 前后端联调基线

### 目标

让前端当前页面能够稳定连接后端开发环境。

### 前端 Tasks

- A2-FE-1 抽离 API base URL 配置
- A2-FE-2 增加环境变量说明与默认值策略
- A2-FE-3 统一请求错误处理
- A2-FE-4 为现有页面补充基础空态和错误态

### 后端 Tasks

- A2-BE-1 增加健康检查接口
- A2-BE-2 配置 CORS
- A2-BE-3 定义统一错误响应格式
- A2-BE-4 增加全局参数校验和异常过滤

### Done

- 前端可请求健康检查和 Swagger
- 请求失败时返回统一错误结构

## A3 数据层初始化

### 目标

用真实业务 schema 替换示例 Prisma schema。

### Tasks

- A3-1 删除示例 `User` 模型
- A3-2 新建阶段 B/C/D1 所需核心模型
- A3-3 初始化首批 migration
- A3-4 建立 Prisma service 与 repository 使用规范

### Done

- 数据库可完成初始化迁移
- 数据模型可以支撑阶段 B/C/D1 开发

## 5. 阶段 B 任务拆解

## B1 登录与鉴权

### 目标

实现家长手机号验证码登录，并返回统一会话上下文。

### 后端 Tasks

- B1-BE-1 实现 `POST /auth/send-code`
- B1-BE-2 实现验证码存储、有效期、频率限制
- B1-BE-3 实现 `POST /auth/login`
- B1-BE-4 实现 JWT 签发与校验
- B1-BE-5 实现 `GET /auth/me`

### 前端 Tasks

- B1-FE-1 调整登录页，不再依赖开发验证码文案
- B1-FE-2 接入验证码发送能力
- B1-FE-3 登录成功后缓存会话信息
- B1-FE-4 处理 token 失效后的跳转逻辑

### Done

- 家长可以完成真实登录
- 登录后前端获得 `accessToken + parentId + familyId`

## B2 家庭主数据初始化

### 目标

在首次登录时自动建立家庭主数据。

### 后端 Tasks

- B2-BE-1 设计 `Parent`、`Family`、`FamilyMembership` 表
- B2-BE-2 首次登录自动创建 `Family`
- B2-BE-3 创建主家长 membership
- B2-BE-4 查询登录用户的家庭上下文

### Done

- 新家长首次登录后可直接进入家庭引导

## B3 第二位家长邀请

### 目标

实现家庭双家长模型。

### 后端 Tasks

- B3-BE-1 实现邀请 token 或邀请码机制
- B3-BE-2 实现 `POST /families/invite`
- B3-BE-3 实现 `POST /families/invite/accept`
- B3-BE-4 限制一个家庭最多两位家长

### 前端 Tasks

- B3-FE-1 新增邀请页
- B3-FE-2 展示邀请状态和失败原因

### Done

- 第二位家长接受邀请后加入同一家庭

## B4 孩子建档

### 目标

完成孩子主数据创建，并自动初始化画像和游戏化档案。

### 后端 Tasks

- B4-BE-1 设计 `Child` 表
- B4-BE-2 设计 `ChildProfile`、`ChildGameProfile`
- B4-BE-3 实现 K12 阶段映射规则
- B4-BE-4 实现 `POST /children`
- B4-BE-5 实现 `GET /children`
- B4-BE-6 实现 `GET /children/:id`

### 前端 Tasks

- B4-FE-1 新增孩子建档页
- B4-FE-2 新增孩子列表/切换能力
- B4-FE-3 在家长首页显示当前孩子上下文

### Done

- 家长完成建档后可看到孩子列表
- 创建孩子时自动生成画像和游戏化档案

## 6. 阶段 C 任务拆解

## C1 学习设置

### 目标

让家长可以为每个孩子配置学习窗口和强度。

### 后端 Tasks

- C1-BE-1 设计 `ChildLearningSettings` 表
- C1-BE-2 实现 `GET /children/:id/settings`
- C1-BE-3 实现 `PUT /children/:id/settings`
- C1-BE-4 增加时间窗口合法性校验

### 前端 Tasks

- C1-FE-1 接入真实设置接口
- C1-FE-2 支持多孩子切换设置
- C1-FE-3 补充输入合法性提示

### Done

- 家长可以保存并读取孩子学习设置

## C2 推送规则引擎

### 目标

在不依赖复杂 AI 的情况下生成可审批的学习建议。

### 后端 Tasks

- C2-BE-1 设计 `LearningPush` 表
- C2-BE-2 建立规则引擎输入结构
- C2-BE-3 按学习设置和复习需求生成建议
- C2-BE-4 生成 `summary/reason/expectedOutcome/contentJson`

### Done

- 后端能够为指定孩子生成待审批建议

## C3 审批流

### 目标

实现家长审批动作和操作留痕。

### 后端 Tasks

- C3-BE-1 设计 `LearningPushActionLog`
- C3-BE-2 实现 `GET /pushes/pending`
- C3-BE-3 实现 `POST /pushes/:id/approve`
- C3-BE-4 支持 `APPROVE / MODIFY / REJECT / POSTPONE`
- C3-BE-5 记录审批操作日志

### 前端 Tasks

- C3-FE-1 接入待审批列表
- C3-FE-2 将 JSON 调试式审批改为产品化交互
- C3-FE-3 支持通过、修改、拒绝、延后
- C3-FE-4 操作完成后刷新列表

### Done

- 家长可完成完整审批流程
- 每次审批动作均有后端记录

## C4 任务流转

### 目标

把审批后的建议转换为孩子端可执行任务，并支持状态推进。

### 后端 Tasks

- C4-BE-1 设计 `LearningTask` 表
- C4-BE-2 审批通过或修改后生成任务
- C4-BE-3 实现 `GET /pushes/tasks/:childId`
- C4-BE-4 实现 `POST /pushes/:id/deliver`
- C4-BE-5 实现 `POST /pushes/:id/complete`
- C4-BE-6 建立任务状态机和状态校验

### 前端 Tasks

- C4-FE-1 接入孩子任务板真实接口
- C4-FE-2 支持按孩子、按日期筛选
- C4-FE-3 支持 delivered/completed 状态推进

### Done

- 孩子端可以拉取待执行任务
- 任务状态流转可追踪

## C5 家长首页收敛

### 目标

把家长首页从跳板页收敛为最小可用仪表盘。

### 前端 Tasks

- C5-FE-1 显示当前家庭和当前孩子
- C5-FE-2 显示待审批数量
- C5-FE-3 显示最近一条任务或结果摘要

### 后端 Tasks

- C5-BE-1 提供首页聚合接口，或复用现有接口组合

### Done

- 家长首页可以作为日常入口使用

## 12. 不进入 A-D1 的内容

以下内容不应混入当前执行范围：

- 阅读、造句、主题口语三类完整场景
- 学习简报历史页
- 画像可视化大屏
- 徽章、等级、每日挑战
- 复杂 AI 推荐和 LLM prompt 编排
- CMS、运营后台、管理后台

原因：

- 这些内容会拉长首轮闭环时间
- 当前目标是尽快形成可验证业务链路

---

## 13. 下一步执行建议

如果要直接进入开发，建议按以下节奏推进：

1. 先完成批次 1，修复工程基线和登录
2. 再完成批次 2，让家长能真正建档和配置
3. 批次 3 交付审批闭环，形成家长侧可演示能力
4. 批次 4 交付孩子端英语最小闭环，形成首个真实产品价值点

建议开发启动顺序：

1. 先实现后端 schema 和鉴权
2. 同步推进前端登录、建档、设置 UI
3. 后端完成推送和任务后立即联调现有前端页面
4. 最后推进英语学习会话和语音评测
