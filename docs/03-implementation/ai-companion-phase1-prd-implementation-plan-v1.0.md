## 2. 当前代码基线结论

### 2.1 前端现状

当前 `frontend/` 已经是独立 uni-app 工程，存在以下已落地页面：

- 登录页
- 家长首页
- 待审批推送中心
- 学习设置页
- 孩子任务板

当前前端已经抽象出一组目标 API：

- `POST /auth/login`
- `GET /children`
- `GET/PUT /children/:id/settings`
- `GET /pushes/pending`
- `POST /pushes/:id/approve`
- `GET /pushes/tasks/:childId`
- `POST /pushes/:id/deliver`
- `POST /pushes/:id/complete`

结论：前端不是空白，但只覆盖了 PRD 中“登录 + 家长审批流 + 学习设置 + 任务状态流转”的窄子集。

### 2.2 后端现状

当前 `backend/` 仍然是通用 Turborepo/NestJS starter，真实业务实现非常少：

- API 只有根路由 hello
- Prisma 只有 `User(email, name)` 示例表
- 没有认证、家庭、孩子、推送、学习、简报、游戏化、AI 记忆相关模块
- 还保留了与当前产品无关的 `apps/web` Next.js 示例工程

结论：后端需要按当前产品重新建立领域模型和接口，不能把现有 starter 视为已实现业务能力。

### 2.3 当前实施的主要阻塞

当前阻塞优先级从高到低如下：

1. 前后端联调基线不一致
2. 后端领域模型缺失
3. 家庭账户与孩子建档未落地
4. 英语学习核心场景未落地
5. AI 记忆、简报、游戏化均未进入实现阶段

## 3. 实施原则

### 3.1 范围原则

- 严格以 PRD 的 Phase 1 英语 MVP 为边界
- Phase 2 语文、Phase 3 数学不进入当前实现
- 月报、成长时间线、社交功能仅保留接口预留，不进入首轮开发

### 3.2 工程原则

- 以当前 `frontend/` 和 `backend/` 为唯一主线，不恢复旧 `apps/*` 业务代码
- 先打通最小闭环，再增加智能化与游戏化能力
- 数据结构先满足产品闭环，避免过早抽象
- 每阶段必须可运行、可联调、可演示

### 3.3 发布原则

- 每个阶段都要求前端有可见页面、后端有真实接口、数据库有真实落表
- 阶段之间允许 UI 简化，但不允许用硬编码伪造业务状态替代真实闭环

---

## 4. Phase 1 范围拆解

PRD 的 Phase 1 实际包含六类能力：

1. 家庭账户体系
2. AI 全人化记忆引擎
3. 英语五大学习场景
4. 智能推送与家长审批
5. 即时学习简报
6. 游戏化激励 v1

结合当前基线，建议按以下顺序实施：

- 阶段 A: 基线与工程对齐
- 阶段 B: 家庭账户与孩子建档
- 阶段 C: 家长端审批与设置闭环
- 阶段 D: 孩子端英语学习 MVP
- 阶段 E: AI 记忆与即时简报
- 阶段 F: 游戏化激励与运营指标
- 阶段 G: 稳定性、性能与上线准备

## 5. 分阶段实施计划

## 5.1 阶段 A: 基线与工程对齐

### 目标

把当前仓库整理成可持续开发的基线，解决“前端有 API 契约，后端没有能力承接”的问题。

### 前端任务

- 抽离环境配置，统一 `dev / test / prod` API 地址
- 修正当前默认 `localhost:3000/api` 与后端端口不一致问题
- 为现有页面补充基础加载态、空态、错误态
- 收敛当前页面中的调试文案和开发口径

### 后端任务

- 明确 `backend/apps/api` 为唯一业务 API 服务
- 删除或冻结 `backend/apps/web` 等当前无关示例工程
- 配置统一全局前缀，例如 `/api`
- 统一端口、CORS、环境变量、Swagger 标题与项目命名
- 建立模块目录骨架：
  - `auth`
  - `family`
  - `children`
  - `pushes`
  - `learning`
  - `briefings`
  - `gamification`

### 数据任务

- 重建 Prisma schema，不再使用示例 `User`
- 初始化基础枚举和迁移目录

### 建议输出

- 可启动的前端
- 可启动的后端
- Swagger 可见真实业务分组
- 最小健康检查接口

### 验收标准

- 前端能稳定请求后端开发环境
- 后端统一返回 JSON 错误结构
- 仓库中不再保留误导性的示例命名和示例路由

## 5.2 阶段 B: 家庭账户与孩子建档

### 对应 PRD

- FR-001 手机号登录注册
- FR-002 家庭成员管理
- FR-003 孩子电子建档

### 目标

先建立“家长 -> 家庭 -> 孩子”的真实主数据，这是后续审批、学习和简报的前置条件。

### 前端任务

- 完成手机号 + 验证码登录流程
- 登录后根据是否已有家庭与孩子，进入不同引导流
- 新增页面：
  - 家庭初始化页
  - 添加孩子页
  - 邀请家长页
  - 孩子列表/切换组件
- 补充家长身份态缓存与登录态失效处理

### 后端任务

- 实现验证码发送与登录接口
- 实现 JWT 鉴权
- 实现首次登录自动创建家庭与主家长账户
- 实现邀请第二位家长加入家庭
- 实现孩子创建、查询、切换
- 实现 K12 阶段自动映射

### 数据模型

建议至少包含以下实体：

- `Parent`
- `Family`
- `FamilyMember`
- `Child`
- `ChildProfile`
- `ChildGameProfile`
- `VerificationCode`
- `AuthSession` 或基于 JWT 的会话黑名单表

### 核心接口

- `POST /auth/send-code`
- `POST /auth/login`
- `GET /auth/me`
- `POST /families/invite`
- `POST /families/invite/accept`
- `GET /children`
- `POST /children`
- `GET /children/:id`

### 测试重点

- 验证码有效期与频率限制
- 家庭最多 2 位家长
- 年级与 K12 阶段映射
- 创建孩子时自动创建 `ChildProfile` 与 `ChildGameProfile`

### 验收标准

- 家长首次登录后可完成建档并看到孩子列表
- 同一家庭可由第二位家长登录查看相同孩子
- 前端登录页不再依赖固定开发验证码文案

## 5.3 阶段 C: 家长端审批与学习设置闭环

### 对应 PRD

- FR-015 AI 智能推送生成
- FR-016 家长审批流程
- 家长端核心功能中的学习设置、审批中心

### 目标

优先完成当前前端已经在做的能力，使现有“审批中心 + 学习设置 + 任务流转”从原型变成真实业务闭环。

### 前端任务

- 完善当前学习设置页
- 补充孩子维度的设置切换
- 简化审批页的调试型 JSON 编辑体验，改为产品化表单操作
- 在家长首页加入：
  - 待审批数量
  - 今日任务概览
  - 最近学习结果摘要

### 后端任务

- 实现孩子学习设置读写
- 实现推送建议生成接口
- 实现待审批列表
- 实现审批动作：
  - 通过
  - 修改
  - 拒绝
  - 延后
- 实现任务状态流转：
  - `PENDING_APPROVAL`
  - `APPROVED`
  - `MODIFIED`
  - `DELIVERED`
  - `COMPLETED`

### 数据模型

建议新增：

- `ChildLearningSettings`
- `LearningPush`
- `LearningPushActionLog`
- `LearningTask`

### 核心接口

- `GET /children/:id/settings`
- `PUT /children/:id/settings`
- `GET /pushes/pending`
- `POST /pushes/:id/approve`
- `POST /pushes/:id/deliver`
- `POST /pushes/:id/complete`
- `GET /pushes/tasks/:childId`

### 调度策略

本阶段不要求完整 AI 能力，可以采用“规则引擎 + 预留 AI 扩展点”：

- 时间窗口命中
- 词量上限
- 日学习时长上限
- 有到期复习任务时优先复习

### 测试重点

- 不同审批动作的状态流转
- 延后时间计算
- 家长修改内容后任务内容保留
- 任务按孩子与日期筛选

### 验收标准

- 家长可以完成从查看建议到批准/修改/延后的整条流程
- 孩子端能拉到已下发任务
- 当前前端已有的审批、设置、任务板页面全部接入真实接口

## 5.4 阶段 D: 孩子端英语学习 MVP

### 对应 PRD

- FR-008 新单词学习
- FR-009 艾宾浩斯间隔复习
- FR-010 多种互动方式
- FR-011 发音练习
- FR-012 情景造句
- FR-013 分级阅读
- FR-014 主题口语对话

### 目标

把 PRD 中“英语五大学习场景”拆成两层交付，先完成最小英语学习闭环，再逐步扩充到完整 5 场景。

### 子阶段 D1: 英语最小闭环

#### 范围

- 单词学习
- 单词复习
- 基础发音跟读

#### 前端任务

- 新增孩子学习主页
- 新增任务开始页、学习会话页、结果页
- 支持题型：
  - 中文选义
  - 单词拼写
  - 基础跟读
- 展示即时反馈：对错、指导、鼓励

#### 后端任务

- 建立英语词库与内容组织模型
- 实现学习会话创建、答题提交、阶段推进
- 实现艾宾浩斯复习节点计算
- 接入语音评测服务，至少返回总分与基础建议

#### 数据模型

- `EnglishWord`
- `ChildWordProgress`
- `LearningSession`
- `LearningSessionItem`
- `PronunciationAttempt`

#### 验收标准

- 一个孩子可以完成“收到任务 -> 学习单词 -> 完成复习 -> 结束会话”
- 会话结束后词汇掌握数据被更新

### 子阶段 D2: 完整英语五场景

#### 增量范围

- 情景造句
- 分级阅读
- 主题口语对话

#### 前端任务

- 新增造句互动页
- 新增阅读材料与阅读理解页
- 新增口语对话页

#### 后端任务

- 造句题生成与评估
- 分级阅读内容匹配与生词沉淀
- 对话主题配置与轮次管理
- 语法纠错与对话总结

#### 验收标准

- 五大学习场景均有最小可用实现
- 孩子端形成真实英语学习闭环

## 5.5 阶段 E: AI 记忆与即时简报

### 对应 PRD

- FR-004 学习数据层记忆
- FR-005 认知风格层记忆
- FR-006 性格情感层记忆
- FR-007 教学策略层记忆
- FR-017 自动生成简报

### 目标

把学习结果沉淀为长期画像，并把家长最能感知价值的“即时简报”做出来。

### 前端任务

- 家长端新增学习简报列表页
- 家长端新增孩子画像摘要页
- 在学习完成后展示会话总结页

### 后端任务

- 建立 `ChildProfile` 的 JSON 画像更新机制
- 将每次学习结果写入长期记忆
- 实现简报生成服务
- 支持按孩子、按日期查询历史简报

### 数据模型

- `ChildProfile`
- `LearningBriefing`
- `ProfileMemorySnapshot` 或画像变更日志表

### AI 能力落点

本阶段 AI 能力建议只做“可解释、可回放”的结构化输出：

- 学习表现总结
- 薄弱点分析
- 鼓励策略推荐
- 下次任务建议

### 验收标准

- 每次学习结束后自动生成简报
- 家长端能看到简报历史
- 画像信息能被后续推送和学习任务复用

## 8. 建议的数据模型主干

为避免后续反复推翻，建议从一开始建立以下主干关系：

`Family -> Parent -> Child -> ChildProfile / ChildGameProfile`

`Child -> ChildLearningSettings -> LearningPush -> LearningSession -> LearningBriefing`

`Child -> ChildWordProgress -> EnglishWord`

`Child -> PointLedger / ChildBadge / AttendanceRecord`

设计原则：

- 画像、推送、学习、简报、游戏化分层建模
- 任务与会话分离，避免“待下发任务”和“已开始学习”混在一张表
- 所有 AI 生成结果都保留结构化字段与原始上下文，方便追踪与调优

## 10. 风险与控制

### 风险 1: 后端范围膨胀

表现：

- 还没打通审批和学习闭环，就试图同时完成完整 AI 平台

控制：

- 阶段 C 前只做规则推送，不做复杂 AI 规划
- 阶段 D1 只做英语最小闭环

### 风险 2: 前端过早产品化

表现：

- 在无真实接口前大量打磨视觉和复杂交互

控制：

- 先保障真实数据闭环
- 视觉增强放到每阶段收尾

### 风险 3: 数据模型频繁推翻

表现：

- 任务、会话、推送、简报责任边界不清

控制：

- 阶段 A 完成领域建模评审
- 所有新表必须说明其所在业务层级

### 风险 4: AI 结果不可追踪

表现：

- 只保留自然语言文本，没有结构化字段

控制：
