# 阶段一：技术共性底座

## 1. 文档定位

本文档定义 `Meglow` 平台的技术共性底座。它关注的是平台统一工程与基础能力，而不是单一业务功能。

## 2. 阶段目标

建立后续所有产品面和业务场景共享的技术基础，包括服务入口、配置管理、数据持久化、基础鉴权、通用校验与错误处理、依赖管理和开发联调能力。

## 3. 本阶段要解决的核心问题

- 没有统一服务入口，后续产品能力无法稳定接入。
- 没有统一配置、依赖和环境能力，开发、测试、部署会持续分裂。
- 没有共用数据与请求处理框架，产品底座和业务实现会重复造轮子。
- 没有基础安全与鉴权骨架，后续角色体系和权限体系难以扩展。

## 4. 本阶段范围边界

### 4.1 纳入范围

- API 服务骨架与统一入口
- 配置、环境和依赖管理
- Prisma 持久化与基础数据模型
- 通用校验、异常过滤、Swagger、CORS
- Docker 级基础依赖
- 基础鉴权框架与通用 Guard 雏形

### 4.2 不纳入当前阶段

- 具体学习业务闭环
- 内容后台产品流程
- AI 平台能力
- 商业化能力

这些能力都依赖本阶段，但不应在本阶段完成定义时混入业务目标。

| 能力项 | 目标形态 | 当前实现 | 状态 | 依据 |

## 6. 当前实现映射

### 6.1 关键代码区域

- `backend/apps/api/src/main.ts`
- `backend/apps/api/src/app.module.ts`
- `backend/apps/api/src/persistence`
- `backend/apps/api/src/common`
- `backend/apps/api/prisma/schema.prisma`
- `backend/docker-compose.yml`

### 6.2 当前代码中的跨阶段混入情况

当前后端应用不是纯底座工程，还混入了阶段二和阶段三能力：

- `AppModule` 已同时加载内容后台、教材内容域和 AI 学伴业务模块。
- Prisma schema 既承载基础模型，也承载业务模型。
- Guard、认证、控制器层已经开始服务不同阶段角色。

因此，当前代码虽然具备阶段一雏形，但不能据此判断阶段一已经完整结束。

## 7. 目标形态 / 当前实现 / 下一步补齐

### 7.1 目标形态

平台应具备统一的技术底座，保证后续产品和业务能力都建立在共享的工程骨架、运行环境、持久化机制和基础安全能力之上。

### 7.2 当前实现

- 统一服务入口、环境依赖、Prisma、Swagger、CORS、通用异常和校验已经具备。
- `PORT`、`JWT_SECRET`、`DATABASE_URL` 等关键运行时配置已开始通过统一 helper 与 schema 校验收口，减少散落读取和空白/非法值带来的运行时歧义。
- `Bearer Token` 提取与缺失令牌异常已开始向 `backend/apps/api/src/common/auth` 收敛，减少 Guard 内部重复请求鉴权样板代码。
- 家长端与后台端 JWT Guard 已开始补齐 payload 结构校验，避免“签名通过但字段不完整”的令牌被直接注入请求上下文。
- 后台角色 Guard 已开始补齐平台级默认权限基线，`SUPER_ADMIN` 可作为后台最高角色覆盖下级后台角色能力。
- 当前家长/后台/开发态认证上下文的提取已开始向统一 request-context helper 收口，避免参数装饰器静默注入空上下文。
- `ApiExceptionFilter` 已开始把默认 `HttpException` 对象响应归一到平台 API 错误结构，减少框架默认异常格式直接外露。
- 查询参数布尔转换、口令 hash 校验等基础工具已开始在 `backend/apps/api/src/common/utils` 收敛，并补齐常见边界值测试、编码格式约束与异常输入防御。
- 家长与管理员鉴权已经分别出现。
- 但统一身份模型、平台级权限治理、分层更清晰的底座边界仍未完成。

### 7.3 下一步补齐

- 继续抽象统一身份与权限边界，避免家长体系和管理员体系长期并列演进。
- 继续收敛基础模块与业务模块的边界，使阶段一能力更容易被阶段二和阶段三复用。
- 明确哪些公共能力应沉淀在 `common`、`persistence`、`config` 等底座层，而非分散在业务模块中。

## 8. 与前后阶段关系

- 前置阶段：无，本阶段是平台起点。
- 为阶段二提供：统一服务骨架、数据与鉴权基础。
- 为阶段三提供：学习业务 API、会话与任务流的底层支撑。

## 9. 当前主要缺口与风险

- 底座与业务模块边界仍然混杂，长期会增加后续平台化难度。
- 统一身份与权限体系未完成，后续角色扩展容易变成多套平行方案。
- 当前阶段缺少专门面向平台底座的文档与验收约束，容易被后续业务代码掩盖。

## 10. 推荐后续动作

### 10.1 推荐后续代码实现

- 梳理家长侧与管理员侧鉴权的共性边界，准备平台统一身份能力。
- 明确底座层公共能力的归属，减少通用逻辑继续散落到业务模块。
- 继续把请求鉴权中的共性前置处理沉淀到 `common/auth`，但保持业务角色语义和错误语义仍由各 Guard 自身控制。

### 10.2 推荐文档补齐

- 后续若引入统一身份、权限或基础事件机制，应优先更新本阶段文档，而不是直接写入业务文档。

### 10.3 推荐验证动作

- 每次底座能力调整后，至少验证 API 启动、环境依赖、Swagger 暴露、基础模块装配是否正常。
- 保持 `docker-compose`、Prisma 和后端启动流程在新文档中的描述与实际命令一致。

## 11. 阶段完成标志

满足以下条件时，阶段一才可视为基本完成：

- 平台具备稳定统一的服务入口和运行环境。
- 基础持久化、配置、校验和异常处理能力可持续复用。
- 统一身份和权限体系达到可扩展状态，而不只是多套登录并存。
- 后续阶段能力可以在不重建底座的前提下稳定扩展。
